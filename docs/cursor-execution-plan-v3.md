# 米伽米工单系统 Cursor 执行计划 V3

## 1. 文档目的

这份文档不是给人开会排期用的，而是给 `Cursor` 持续执行产品优化时使用的操作蓝图。

核心思想：

- 不按“产品、设计、研发、测试”这种人力协作方式拆。
- 改为按 `Cursor 能独立完成的最小可验证批次` 来拆。
- 每一批都必须能直接落到仓库代码、页面、文案、埋点、验证动作上。
- 每一批都要明确哪些事情 `Cursor 可直接做`，哪些事情 `必须你拍板`，哪些事情 `必须依赖真实环境`。

适用场景：

- 你准备让 Cursor 连续多轮改造这个项目。
- 你希望每次只做一小批、做完能验证、不把项目搞乱。
- 你希望计划天然适合 agent 执行，而不是适合人讨论。

---

## 2. Cursor 执行原则

## 2.1 拆分原则

Cursor 最适合执行的是：

- 单一目标明确
- 影响范围可控
- 改动文件可枚举
- 验收条件可自动或半自动验证
- 失败后容易回滚

因此本项目优化不应一次性要求：

- “全面重构工作台”
- “重做整个协作系统”
- “统一所有逻辑和文案”

而应该拆成这种粒度：

- “收敛成员默认入口和侧边栏命名”
- “把提醒卡片改成可直接写进展”
- “给管理驾驶舱增加一个待干预模块”
- “新增企业微信健康状态页”

## 2.2 批次定义

每个批次必须包含 6 个固定字段：

1. `目标`
2. `Cursor 直接做什么`
3. `需要你确认什么`
4. `会动哪些文件`
5. `怎么验证`
6. `完成标准`

没有这 6 个字段的任务，不建议直接交给 Cursor 连续执行。

## 2.3 三类任务分类

### A 类：Cursor 可直接完成

特点：

- 页面结构调整
- 文案统一
- 组件抽取
- 小范围数据查询重构
- 埋点补充
- 加基础验证

这类任务可以直接让 Cursor 实施。

### B 类：Cursor 做，用户拍板

特点：

- 导航命名变更
- 页面职责合并
- 交互路径调整
- 新增字段展示方式
- 默认首页逻辑变化

这类任务应先让 Cursor 产出方案，再由你确认后实施。

### C 类：Cursor 写代码，但依赖真实环境验证

特点：

- 企业微信通知
- Cron 触发效果
- 登录链路
- 外部 API
- 真实数据质量

这类任务可由 Cursor 完成代码部分，但最后需要你配合线上或本地真实环境验证。

## 3. 本项目的正确执行顺序

对于这个项目，最合理的执行顺序不是“先想完整产品，再统一开发”，而是按以下顺序：

1. 先收敛信息架构与页面心智。
2. 再优化高频动作链路。
3. 再强化管理处置能力。
4. 再建设企业微信健康治理。
5. 最后再做规则、埋点和持续优化。

原因：

- 现在最大的浪费不是“缺功能”，而是“已有功能不够顺”。
- 先做结构收敛，后面改动才不会反复推翻。
- 先把主路径做顺，后面的管理和运营能力才有真实数据价值。

## 4. 执行总图

建议拆成 7 个批次执行：

| 批次 | 名称 | 类型 | 建议是否可直接开工 |
|------|------|------|--------------------|
| Batch 0 | 基线梳理与约束确认 | B 类 | 是 |
| Batch 1 | 导航与入口收敛 | B 类 | 需要你先确认 |
| Batch 2 | 工作台与我的任务收敛 | B 类 | 需要你先确认 |
| Batch 3 | 提醒中心可执行化 | A/B 类 | 可先做低风险部分 |
| Batch 4 | 快速更新能力统一 | A 类 | 可直接开工 |
| Batch 5 | 管理驾驶舱处置化 | B 类 | 先出方案再改 |
| Batch 6 | 企业微信健康中心 | A/C 类 | 可开发，需真实环境验证 |
| Batch 7 | 埋点、指标与文案清理 | A 类 | 可直接开工 |

说明：

- `Batch 0` 必须先做，它是后面所有 Cursor 执行的上下文基准。
- `Batch 1-2` 是结构性调整，必须先拍板再实施。
- `Batch 3-7` 才适合连续编码推进。

## 5. 逐批次执行方案

## 5.1 Batch 0：基线梳理与约束确认

### 目标

让 Cursor 在后续改造中不反复猜测页面职责和目标，先把“哪些页面是谁的主入口、哪些功能是高优先级”固定下来。

### Cursor 直接做什么

- 读取并归纳当前页面结构。
- 输出当前主导航职责表。
- 输出页面合并建议和低风险改名建议。
- 标记哪些改动属于结构级改动，哪些属于局部优化。

### 需要你确认什么

- 成员默认首页是否继续固定为 `工作台`
- 是否保留独立 `我的任务`
- `待你回应` 是否改名为 `待处理`
- `成员与钉钉` 是否统一改为企业微信相关命名

### 会动哪些文件

这一批原则上只生成或更新文档，不直接动产品代码。

### 怎么验证

- 你看完后能明确回答“成员今天先去哪一页”
- 你看完后能明确回答“我的任务和工作台是否保留双入口”

### 完成标准

- 有一份确认后的页面职责表
- 有一份确认后的改造边界清单

### 建议给 Cursor 的指令模板

```md
请先只做结构梳理，不改代码。
阅读当前首页、工作台、我的任务、问题列表、提醒中心、驾驶舱、侧边栏。
输出：
1. 当前页面职责表
2. 重叠能力列表
3. 低风险可直接改动项
4. 需要我确认的决策点
```

---

## 5.2 Batch 1：导航与入口收敛

### 目标

让用户进入系统后更容易理解“我先去哪、每个页面是做什么的”。

### Cursor 直接做什么

- 调整侧边栏命名。
- 调整默认首页跳转逻辑。
- 清理明显过时或误导性的文案。
- 如已确认，可合并某些低价值重复入口。

### 需要你确认什么

- 是否保留 `我的任务` 作为一级导航
- `待你回应` 的最终命名
- `看板总览` 是否改为 `管理驾驶舱`

### 可能涉及的文件

- `src/app/page.tsx`
- `src/components/sidebar.tsx`
- `src/app/(main)/home/page.tsx`
- 相关页面标题与描述文件

### 怎么验证

- 登录后角色跳转符合预期
- 侧栏名称与页面内容一致
- 页面数量减少或心智更清晰，但没有破坏现有流程

### 完成标准

- 一个新用户只看导航就能理解主路径
- 历史“钉钉”命名清理完毕

### 风险提示

这是结构性改动，改之前必须先确认命名和保留策略。

### 建议给 Cursor 的指令模板

```md
基于已确认的信息架构，实施导航与入口收敛。
要求：
1. 修改默认首页跳转和侧边栏命名
2. 不做大范围页面删除
3. 保持现有功能可用
4. 完成后说明哪些地方的用户心智变清晰了
5. 跑 lint 检查受影响文件
```

---

## 5.3 Batch 2：工作台与我的任务收敛

### 目标

把成员每天最核心的动作集中到一个稳定入口里，减少页面来回跳。

### Cursor 直接做什么

- 对比 `工作台` 和 `我的任务` 的功能重叠。
- 将“今日建议先处理”“今日待同步”“我负责的活跃事项”收敛到一个更清晰的展示结构。
- 保留另一个页面作为深度视图，或者弱化其独立性。

### 需要你确认什么

- 最终是否保留 `我的任务` 独立页面
- 工作台是否承担“今日主操作台”的定位

### 可能涉及的文件

- `src/app/(main)/home/page.tsx`
- `src/app/(main)/my-tasks/page.tsx`
- `src/components/my-tasks-client.tsx`
- `src/lib/workbench-queries.ts`

### 怎么验证

- 成员进入工作台后，不需要再额外理解两个近似页面
- “今日该更新什么”可以在一个核心区域中完成

### 完成标准

- 工作台变成成员主操作台
- `我的任务` 的存在价值清晰可解释

### 技术备注

这一批还可以顺手处理 `my-tasks` 页面里逐条调用 `issueHasUpdateToday()` 的性能问题，把“今日是否更新”改成批量计算方式。

### 建议给 Cursor 的指令模板

```md
请重构工作台与我的任务的职责边界。
目标：
1. 强化工作台作为成员主入口
2. 减少和我的任务之间的重复信息
3. 优先复用现有组件
4. 如发现明显性能问题，一并小幅修复
5. 不引入新的复杂状态管理
```

---

## 5.4 Batch 3：提醒中心可执行化

### 目标

让提醒从“告诉你发生了什么”变成“你可以直接处理事情”。

### Cursor 直接做什么

- 在提醒卡片上增加快捷操作按钮。
- 支持从提醒直接触发快速更新。
- 让提醒处理后给出即时反馈。
- 为不同提醒类型做不同主操作。

### 需要你确认什么

- 每类提醒的主操作是什么
- 是否允许在提醒页直接改状态
- 是否允许在提醒页直接申请改截止时间

### 可能涉及的文件

- `src/app/(main)/reminders/page.tsx`
- `src/components/reminders-client.tsx`
- `src/actions/reminders.ts`
- `src/actions/issues.ts`
- 可能新增一个共用的“快速更新弹层”组件

### 怎么验证

- 用户在提醒页可直接处理至少一类高频提醒
- 提醒处理完成后，状态能在界面上即时反映
- 没有破坏原有提醒已读逻辑

### 完成标准

- 至少实现一种“提醒直接推进”的路径
- 最好优先实现 `今日待同步 -> 直接写进展`

### 执行建议

不要第一版就把所有提醒类型都做完。

先做：

- `今日待同步`
- `连续 3 天未更新`

等交互跑顺后，再扩展到逾期、待确认等类型。

### 建议给 Cursor 的指令模板

```md
请把提醒中心改造成可执行页面，但先做最小可用版本。
要求：
1. 优先支持“今日待同步”直接写进展
2. 保持现有提醒列表可读性
3. 尽量复用问题详情页已有更新逻辑
4. 如需新增组件，做成通用组件，后续可给工作台和我的任务复用
```

## 5.5 Batch 4：快速更新能力统一

### 目标

把“写进展”做成全系统统一动作，而不是每个页面各写一套。

### Cursor 直接做什么

- 抽一个统一的快速更新组件。
- 在工作台、我的任务、提醒中心、问题详情中复用。
- 统一默认字段、默认状态、提交反馈。
- 必要时补充服务端 action 的共用能力。

### 需要你确认什么

- 快速更新是否必须包含 3 段式模板：
  - 今天做了什么
  - 当前卡点
  - 下一步计划
- 是否允许直接切换状态

### 可能涉及的文件

- `src/components/issue-updates-client.tsx`
- `src/components/my-tasks-client.tsx`
- `src/components/reminders-client.tsx`
- `src/components/issue-detail-client.tsx`
- `src/actions/issues.ts`

### 怎么验证

- 四个入口的写进展交互一致
- 提交后刷新逻辑一致
- 状态和时间线展示不冲突

### 完成标准

- 同一种更新动作在不同页面表现一致
- 代码层面不再有多套重复更新 UI

### 执行收益

这一批对 Cursor 非常友好，因为它属于典型的“抽组件 + 收敛逻辑 + 统一交互”类型。

### 建议给 Cursor 的指令模板

```md
请统一项目中的“写进展”交互。
目标：
1. 抽取一个通用快速更新组件
2. 在工作台、我的任务、提醒中心、问题详情中复用
3. 保持现有服务端动作兼容
4. 完成后检查是否出现重复代码可继续删减
```

## 5.6 Batch 5：管理驾驶舱处置化

### 目标

让驾驶舱不只是看数据，而是直接推动管理动作。

### Cursor 直接做什么

- 在驾驶舱新增“待干预事项”区块。
- 把高风险事项、通知失败、成员负载异常串起来展示。
- 提供跳转动作或直接处理动作入口。

### 需要你确认什么

- 管理者最想优先看到的 3 类问题是什么
- 是否允许在驾驶舱直接做转派或提醒重发
- 第一版是否只做“查看和跳转”，先不做直接修改

### 可能涉及的文件

- `src/app/(main)/dashboard/page.tsx`
- `src/lib/dashboard-queries.ts`
- `src/actions/dashboard.ts`
- `src/actions/notifications.ts`

### 怎么验证

- 管理者能在驾驶舱中更快定位风险问题
- 至少新增一个“从驾驶舱直接到具体处理页面”的高价值路径

### 完成标准

- 驾驶舱上有清晰的“今日必须干预”区域
- 风险数据不只是统计，而是带动作建议

### 第一版建议

先做轻版本：

- 待干预事项列表
- 每条附带“查看问题”“查看成员”“查看通知异常”

先别急着做复杂的直接编辑动作。

### 建议给 Cursor 的指令模板

```md
请把管理驾驶舱改成更可执行，但第一版只做轻量处置化。
要求：
1. 新增“今日必须干预”模块
2. 复用现有查询能力，少写新逻辑
3. 优先做高风险问题和通知异常两个方向
4. 第一版以跳转处理为主，不引入复杂原地编辑
```

## 5.7 Batch 6：企业微信健康中心

### 目标

把“通知没到”从隐形问题变成系统内显性问题。

### Cursor 直接做什么

- 新增一个健康状态页面或模块。
- 汇总成员接入状态、通知成功率、失败类型、失败用户。
- 增加测试入口或操作引导。

### 需要你确认什么

- 页面放在 `members` 下还是单独做 `wecom-health`
- 第一版是否展示企业可信 IP 说明
- 是否允许展示失败原因原文

### 可能涉及的文件

- `src/app/(main)/members/page.tsx`
- 或新增 `src/app/(main)/wecom-health/page.tsx`
- `src/actions/members.ts`
- `src/actions/notifications.ts`
- 可能补充新的查询函数

### 怎么验证

- 系统内能看出谁没绑定 `wecom_userid`
- 系统内能看出最近通知失败最多的问题点
- 管理员能知道下一步该去哪里修

### 完成标准

- 新增一个可用的企业微信健康视图
- 至少覆盖成员绑定、发送成功率、失败列表三项

### 真实环境依赖

这一批 UI 和查询逻辑可以让 Cursor 直接开发，但最终仍需要真实数据验证：

- 是否真有失败样本
- 企业微信字段是否齐全
- Cron 产生的数据是否完整

### 建议给 Cursor 的指令模板

```md
请新增企业微信健康中心，优先做一个可上线的观察页。
要求：
1. 聚焦可观测性，不追求一步到位
2. 先展示成员绑定、通知成功率、失败记录
3. 页面文案要让管理员知道“这是什么问题、下一步怎么修”
4. 完成后说明哪些部分还需要真实环境验证
```

## 5.8 Batch 7：埋点、指标与文案清理

### 目标

让后续优化不靠感觉，而有真实数据和统一语言。

### Cursor 直接做什么

- 统一导航、标题、空状态、按钮文案。
- 为关键路径补埋点。
- 补充必要的注释和验证文档。

### 需要你确认什么

- 是否已有埋点体系
- 事件命名约定是什么
- 是否先只做前端事件埋点

### 可能涉及的文件

- 各主页面与主组件
- `@vercel/analytics` 相关接入点
- 项目文档

### 怎么验证

- 页面文案前后一致
- 关键动作能被记录
- 后续能比较改版前后行为变化

### 完成标准

- 关键路径至少有基础事件记录
- 文案明显统一，不再出现历史体系混用

### 建议给 Cursor 的指令模板

```md
请做一轮低风险的文案和埋点清理。
要求：
1. 不改业务逻辑
2. 统一导航、页面标题、按钮文案
3. 给工作台、提醒、快速更新、驾驶舱等关键动作补基础埋点
4. 输出埋点事件清单
```

## 6. 最适合 Cursor 的执行节奏

不建议一句话让 Cursor：

`把整个系统优化一遍`

建议采用下面这种节奏：

1. 先让 Cursor 只做阅读和输出改造方案。
2. 你确认决策点后，再让 Cursor 改一小批。
3. 每一批改完后，要求 Cursor：
   - 说明改了什么
   - 跑最小验证
   - 列出剩余风险
4. 再进入下一批。

## 6.1 推荐单轮工作时长

每轮最好控制在：

- 1 个明确主题
- 1 到 4 个核心文件
- 1 组可验证结果

太大时会出现的问题：

- Cursor 改动面过大
- 用户难以 review
- 容易把多个决策混在一起

## 6.2 推荐单轮输出格式

你每次给 Cursor 的任务，建议要求返回：

- 改动摘要
- 影响文件
- 已完成验证
- 未解决问题
- 下一轮建议

## 7. 推荐实际执行顺序

如果你现在就开始让 Cursor 干，我建议这样排：

### 第 1 轮

- 做 `Batch 0`
- 产出页面职责表和决策点

### 第 2 轮

- 做 `Batch 1`
- 先收敛导航命名和默认入口

### 第 3 轮

- 做 `Batch 2`
- 收敛工作台和我的任务

### 第 4 轮

- 做 `Batch 4`
- 先统一快速更新能力

### 第 5 轮

- 做 `Batch 3`
- 把提醒页接上快速更新能力

### 第 6 轮

- 做 `Batch 5`
- 增加驾驶舱“待干预”模块

### 第 7 轮

- 做 `Batch 6`
- 补企业微信健康中心

### 第 8 轮

- 做 `Batch 7`
- 清文案、加埋点、补说明

说明：

- 我把 `Batch 4` 放在 `Batch 3` 前面，是因为先统一快速更新组件，再接提醒页，会更稳、更省返工。

---

## 8. 不建议 Cursor 直接一口气做的事

下面这些不建议一次性丢给 Cursor：

- 同时重构工作台、我的任务、提醒中心、详情页
- 一口气改完所有提醒类型逻辑
- 一次性重做整个驾驶舱数据结构
- 在没有真实环境的情况下承诺企业微信链路完全可用
- 在没有埋点前就断言改版效果一定更好

正确方式是：

- 先做最小可用版本
- 用真实反馈修下一轮

---

## 9. 结论

如果现在是用 Cursor 来推进这个项目，最好的计划方式不是“像团队项目管理那样列人天和分工”，而是：

- 先定结构性决策点
- 再按最小可验证批次持续改
- 每批只做一个核心主题
- 每批都要求可验证、可回滚、可继续接力

对这个项目来说，最佳的 Cursor 执行顺序是：

1. 先收敛导航和页面职责
2. 再统一快速更新
3. 再让提醒可执行
4. 再做管理处置化
5. 最后补企业微信健康和埋点

这样安排，既符合 Cursor 的执行方式，也最不容易把当前已经能跑的系统改乱。
